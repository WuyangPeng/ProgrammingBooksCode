It seems like all scripts are being rejected. What happened and how can we fix the problem?
Rejected Scripts

To understand script rejections, you have to know that there is a "pecking order" among members of the project. They are ordered by their user id. The top dog, usually the administrator, is the one with the lowest id (often 0).

A conflict happens when two scripts miss each other. (That's why it's a good idea to check your e-mail before doing a check-in. This doesn't always work, because there are sometimes large delays in e-mail.)

Suppose user A with id 0 does a check-in and sends the script to user B. User B, with id 1, in the meanwhile, does a separate check-in, before obtaining the script from A. He sends his script to A.

What happens is the following:

  • User A gets a script from B and, since his id is 0, he wins. Script from B is rejected. He doesn't have to do anything other than delete the script from B.
  • User B gets the script from A and unpacks it. Because of the conflict, his previous changes are automatically undone. They are not lost, however. All the files that were changed by his last check-in are checked out again, so that he can try checking them in again. That's what he should do: after accepting the synch from A, he should check in his files again. In some cases there might be overlap between changes from A and B. In such an event B still has the opportunity to modify his files (merge the changes) before checking them in.
  • Suppose that there is user C and he gets the same scripts, but the script from B comes before the script from A. He first unpacks the script from B and continues happily. Now the script from A comes in and knocks out the losing script from B. The changes made by B are automatically undone--a dialog is displayed giving details of the restoration. User C can continue as if nothing had happened.
Are my problems due to lost ACK scripts?
Lost Acknowledgments
Occasional loss of acknowledgment scripts does not impair the working of the system. It only temporarily lengthens the lineage list sent with every script. Eventually, when a new version is voted in, the lineages shrink back to normal.

Acknowledgments work like this: every voting member, after receiving a script, produces an acknowledment addressed to the script's creator. After collecting all the acknowledgments, the creator sends one more final ack to everybody, thus blessing the version as "Confirmed."

A lineage of a script is the list of tentative versions upon which this script depends, ending with a confirmed version, whose acceptance has been confirmed by everybody.

I get a "checksum mismatch" error
Checksum Mismatch
Do not manipulate project files outside of Code Co-op. Especially do not change their read-only attribute. If you inadvertently do that, the next attempt to check the file out or synchronize it will fail. To fix the problem, run Repair from the Project menu.
Why are some versions marked "tentative"?
Tentative Versions

A version is marked "tentative" if you haven't received all the acknowledgments scripts for it yet. There's nothing to be worried about. Even if a few acknowledgments are missing for a given script, some other script after that will be confirmed, thus automatically confirming all previous script. If the list of "tentative" versions doesn't seem to shrink for a long time, it means that there is a member of the project who is not responding to any scripts at all. Find out who it is, by double-clicking on some of the older "tentative" versions. At the very end of the listing, you'll see the line, "Expecting acknowledgments from:". There's your culprit!

If you know for sure that this person is no longer a member of the project, because he or she defected from it, ask somebody else to re-send this person's defect script to you.

If this person left for vacation, see the next article.

The admin left for vacation!
Going on Vacation

If a member of the project wants to go on vacation, he or she should change their status to Observer. This way nobody will be waiting for the acknowledgments from that person. After coming back from vacation, he or she will have to unpack and accept a whole bunch of scripts, and then change the status back to Voting member. (All this status changing is done through the Members dialog in the Project menu.)

What if somebody forgot to change their status? --The administrator can change it for them. What if the administrator left for vacation? --You have to invoke the emergecy administrator election. Look for it under Administrator in the Project menu.

I had a trial version of 1.0
Now I can't get the trial version of 2.1 to work.
Expired Trial Period

In order to restart your trial period, you have to defect from all your projects, uninstall Code Co-op and install it again. The uninstall is available from your Control Panel, under Add/Remove Programs. Older versions of uninstall might not be thorough enough. In that case, after installing the newest version, uninstall it again and repeat the installation.

Of course, if you have a valid license, none of this is necessary. The upgrade is automatic when you run the setup of a newer version of Code Co-op.

Are scripts rejected because some scripts were lost in the mail?
Missing Scripts

Scripts are not rejected because some scripts were lost. When a script is lost and co-op realizes this, it will display the missing script in the inbox (it knows its script id from the lineage of a later script) and mark it as missing. Ask some other member of the project to re-send a copy of the missing script to you. To re-send a script, select it in History view and click on the envelope button.

I'd like to know if the file is checked out by somebody else.
Check-out Notification

Some version-control systems notify other users when you check out a file. Others let you lock a file, so that nobody else can check it out. Code Co-op doesn't do it for several reasons. First of all, it's not very practical in a slow, distributed environment, where notification often travel by email. Second of all, there is no central server, so there is no sure way of locking a file, short of waiting for the lock acknowledgment from everybody. Finally, Code Co-op relies on conflict resolution rather than the more obtrusive conflict prevention. Which doesn't mean that in future releases we won't implement optional check-out notification, especially on local-network systems.

What's binary type?

File Types

The first time you add a file with a new extension, Code Co-op asks you about the associated type. You have a choice of header, source, text and binary. Binary types behave differently than others. They are diffed using our custom binary differ, so their storage and exchange is optimal. If you double-click on a binary file in Code Co-op, instead of firing the Differ, the program will use your shell's file-type association to call the appropriate application. Binary files cannot be merged, so when there is a conflict, Code Co-op makes a backup of your current version of a file using the name "Copy of..."

I get the Windows Messaging Setup Wizard when I try to "Get Mail" in dispatcher.
Windows Messaging Setup

If Windows pops up a Windows Messaging Setup dialog when you try to Get Mail in the dispatcher, and you are presented with a list that doesn't have Internet Mail in it, try the following:

  1. Click on "Start", "Settings", "Control Panel" and double-click "Add/Remove Programs",
  2. Click on the "Windows NT Setup" tab,
  3. Highlight "Windows Messaging" in the Components listbox,
  4. Click the "Details" button and make sure "Internet Mail" is checked and then click "OK", "OK".
Can I undo automatic merge?
Undo Merge

On some occasions you might not like the results of automatic merge. You can always correct the results by editing the file inside the differ. But if you really want to see the original pre-merge file with your own edits, there is a back-door approach. First, find out the project id (in the Projects area)and the file's global id (last column in File Area). Then, go to the Code Co-op's data directory for this particular project. In most cases it's in the co-op\database\n subdirectory of your project, where n is the project id. The file whose name is formed from the global id of the file in question followed by the extension .bak is your pre-merge copy. Don't edit this file! It's part of Code Co-op's database. Just copy it to a safe place and work with it. Remember, making any changes in Code Co-op's data directory will result in database corruption and possible loss of the project.

My project database is lost beyond repair!
Project Beyond Repair

Despite our best efforts, there is a non-zero chance that you will lose your project. Disks fail, files get overwritten or inadvertently deleted. If all your efforts to repair the project fail, your only resort is to defect and re-enlist. Sometimes you can't even defect--ask the administrator to remove you from the project. If you're the administrator, ask somebody to perform the emergency admin election. As long as there are other project members, your project is safe. Code Co-op is a very reliable backup system. Just join the project again, and you'll get a fresh copy of all the files. Moreover, if you want, you can receive the whole history of the project from one of the members. History>Export produces a file containing history. This file can be moved to your machine and you can do History>Import to load it.

BTW, having an additional "observer" enlistment may be a good backup strategy.
